# 首批工程 PR 拆分建议

## 总原则
- 第一批 PR 不要按“人”拆，而要按“可验证的系统能力”拆。
- 每个 PR 都必须独立可 review、可运行、可验证。
- 第一批 PR 的目标不是把系统做完，而是把 MVP 主链路打通。

## PR-1
### 标题
`feat: add P0 data loaders and schema validation`

### 目标
- 建立五张 `P0` 表的统一读取方式
- 固化 schema 校验
- 让后续 retrieval 和 pipeline 不再直接依赖原始 csv 细节

### 范围
- `product_sku`
- `review_feedback`
- `trend_signal`
- `ingredient_knowledge`
- `compliance_rule`

### 交付定义
- 能读取当前基线交付包
- 能输出统一数据对象
- 能在字段缺失、主键重复、时间格式错误时失败并报错

### 负责人建议
- 工程 A 主做
- 你 review schema、字段语义和 batch 版本规则

## PR-2
### 标题
`feat: add retrieval index for product ingredient trend and compliance`

### 目标
- 把五张 `P0` 表转成可检索证据层
- 支撑后续 QA pipeline 的证据召回

### 范围
- 精确检索：产品、成分、法规条目
- 关键词检索：趋势词、功效词、风险词
- 返回统一 evidence item 格式

### 交付定义
- 给定 query，可以返回证据列表
- 每条证据包含 `source_table`、`source_id`、`score`、`snippet`
- 合规问题必须能召回 `compliance_rule`

### 负责人建议
- 工程 A 或 工程 B 均可
- 如果 A 做 loader，则这个 PR 建议 A 继续做，减少上下文切换

## PR-3
### 标题
`feat: add grounded QA orchestration baseline`

### 目标
- 打通最小问答链路
- 不追求模型训练，先做 grounded baseline

### 范围
- query intent classification
- retrieval routing
- evidence assembly
- answer template / prompt template
- risk note / compliance note

### 交付定义
- 至少支持四类问题：
- 科学护肤问答
- 成分风险问答
- 趋势相关问答
- 商品推荐解释问答
- 输出必须带 evidence ids

### 负责人建议
- 工程 B 主做
- 你定义意图分类和回答边界

## PR-4
### 标题
`feat: add MVP offline benchmark and evaluation runner`

### 目标
- 给团队一个共同的“好坏判断标准”
- 避免后续讨论停留在主观感觉

### 范围
- 冻结 30 到 50 个 MVP benchmark questions
- 定义 answer format
- 定义 grounding / compliance / trend freshness 指标
- 输出简单评测报告

### 交付定义
- 可以对同一版本 pipeline 重复跑评测
- 可以比较版本前后差异

### 负责人建议
- 你主做 benchmark 和指标定义
- 工程 B 配合接入 runner

## 推荐顺序
1. `PR-1` 先做
2. `PR-2` 紧接着做
3. `PR-3` 在 `PR-2` 基础上做
4. `PR-4` 和 `PR-3` 可部分并行

## 你的位置
- 你不必亲自写所有工程代码，但你必须卡住三件事：
- schema 和数据边界
- 问答边界和风险边界
- 评测标准和版本门禁

## 适合你发给团队的一句话
“第一阶段先不扩范围，先基于五张 `P0` 表把 retrieval-grounded MVP 打通；所有实现按独立 PR 推进，并统一挂到 tracking issue 下。”
