# Task Package Template

## Metadata
- Task ID:
- Module:
- Owner Type: `dev-agent / engineer / qa-owner / architecture-control`
- Status: `draft / ready / in_progress / review / blocked / done`
- Related Contract Docs:
- Related Benchmark / Backtest:

## Goal
- What this task is supposed to achieve.

## Scope
- What is included.

## Non-Goals
- What must not be changed in this task.

## Inputs
- Required docs, schemas, code modules, datasets, or upstream contracts.

## Required Outputs
- Code / doc / dataset / report outputs expected from the executor.

## Contract Constraints
- Interfaces or field names that must stay stable.

## Execution Boundary
- What the executor may change freely.
- What the executor must not change without architecture review.

## Implementation Notes
- Practical notes for the executor.

## Backtest Requirements
- Cases that must be run.
- Metrics or qualitative checks that must be reported.

## Acceptance Criteria
- Conditions required for merge or handoff.

## Evidence Required In Return
- Changed files
- Run results
- Backtest summary
- Open risks

## Suggested Kickoff Instruction
Use or adapt the block below when opening a dedicated execution thread.

```text
Task ID:
Goal:
Scope:
Non-goals:
Required outputs:
Backtest requirements:
Constraints:
Return format:
- changed files
- what was implemented
- backtest result
- open risks
```
