# 两周迭代 SOP（v1）

## 节奏定义
- 迭代周期：14天
- 目标：每轮产出一个“可评估、可灰度”的版本
- 版本命名：`beauty-v{major}.{minor}`（如 `beauty-v1.2`）

## D1-D2：范围冻结
- 明确本轮业务范围（例如“敏感肌修护推荐”）。
- 冻结数据预算（SKU/评论/趋势词数量）。
- 输出：`scope.md`、`data_budget.yaml`。

## D3-D5：数据抽取与清洗
- 按分层规则抽样。
- 清洗：去重、缺失修复、枚举归一化、异常检测。
- 输出：`cleaned/*`、`data_quality_report.md`。

## D6：知识构建
- 更新成分知识、规则库、趋势词。
- 生成可检索知识包（供 RAG/规则模块使用）。
- 输出：`knowledge_pack_vN`。

## D7-D9：蒸馏与训练
- Teacher 生成样本（受规则约束）。
- 合规过滤 + 置信度打标。
- Student 蒸馏训练（含旧样本回放，防遗忘）。
- 输出：`distill_dataset_vN`、`student_ckpt_vN`。

## D10：评测
- 运行固定测试集 + 回放集 + 本轮新增趋势集。
- 输出：`eval_report_vN`（包含 correctness/completeness/safety/latency）。

## D11：灰度发布
- 小流量（建议 10%）验证。
- 监控违规率、风险词触发率、响应时延。
- 输出：`gray_report_vN`。

## D12-D14：复盘与回流
- 汇总失败样本，进入 hard-case 池。
- 输出下一轮优化点和预算申请。
- 输出：`retrospective_vN`、`hard_cases_vN`。

## 角色分工建议
- 架构负责人：范围、门槛、发布决策。
- 数据负责人：采样、清洗、质量报告。
- 模型负责人：蒸馏、训练、回归评测。
- 业务负责人：样本验收、效果反馈。
- 合规负责人：规则更新、违规审查。

## 关键纪律
- 不允许跳过 D10 评测直接上线。
- 不允许使用未备案数据源。
- 不允许删除历史评测结果（需可追溯）。
