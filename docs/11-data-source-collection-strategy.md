# 11 - Data Source And Collection Strategy

## 1. 目标
- 把数据层上游从“已有数据清单”推进到“可持续采集与更新策略”。
- 让数据供给不只服务一次性 baseline，而是服务后续 batch、evaluation、distillation 的连续循环。
- 明确哪些数据值得优先投入，哪些数据先保持轻量化处理。

## 2. 为什么这件事关键
- 当前 MVP 的工程实现依赖五张 `P0` 表，而这五张表都不是天然稳定供给。
- 如果没有明确的数据源和采集策略，后续 batch 只能靠临时拉数，工程接口和评测口径都会漂移。
- 对这个项目而言，先稳定数据收集和更新机制，等于先稳定了一半的系统边界。

## 3. 当前数据层状态
- 已有第一轮 `P0`：`product_sku`、`review_feedback`、`trend_signal`、`ingredient_knowledge`、`compliance_rule`
- 已有第一轮 `P1`：`review_feedback_raw`
- 已完成第一轮清洗、合规结构化和 batch 交付
- 但“源头如何持续更新”目前仍然是经验性讨论，尚未形成正式策略

## 4. 建议的数据源分层

### 4.1 Core Internal
- 商品主数据
- 评论与口碑标签
- 趋势热词与内容热度

这是最关键的数据源层，因为它直接决定商品、趋势、反馈三张核心表的更新能力。

### 4.2 Public Structured
- 成分标准信息
- 法规与监管公告
- 公开原料目录
- 功效规范和标签规范

这是科学解释和合规判断的基础层，更新频率不需要很高，但必须可靠。

### 4.3 Model-Assisted
- 弱画像标签
- 评论摘要与标签补全
- 查询意图标签
- 趋势主题聚类

这层不直接作为原始事实来源，而是作为结构化增强层。

## 5. 按表梳理的数据源与采集策略

| 目标表 | 主数据源 | 采集策略 | 更新建议 | 重点风险 |
|---|---|---|---|---|
| `product_sku` | 内部商品库 | 固定批次导出，按类目/价格带/品牌梯度分层抽样 | 周更 | 类目漂移、字段定义不一致 |
| `review_feedback` | 内部评论接口 | 先抽样、再标签化、原文降级到 `P1` | 日更/双周打包 | 原文过多、评分缺失、情感标签噪声 |
| `trend_signal` | 内容/热词接口 | 固定时间窗抓取 + 增长率筛选 + 趋势过采样 | 日更 | 时间窗不一致、热点短暂波动 |
| `ingredient_knowledge` | 成分库、词典、文献整理 | 先标准成分表，再补功效/机理/风险字段 | 周更或月更 | 同义词混乱、证据等级不统一 |
| `compliance_rule` | 法规文件、目录、公告 | 先条款与目录结构化，再补义务 taxonomy | 月更 + 事件触发 | 条款粒度不一致、法规修订追踪困难 |

## 6. 采集策略的优先级建议

### 6.1 第一优先级
- `product_sku`
- `review_feedback`
- `trend_signal`

原因：
- 它们直接决定 MVP 的“商品理解 + 用户反馈 + 趋势响应”三条主链路。
- 这三者稳定后，工程 A 和工程 B 的接口才能稳定。

### 6.2 第二优先级
- `compliance_rule`
- `ingredient_knowledge`

原因：
- 它们决定系统的可信度和安全边界。
- 不需要高频更新，但必须形成规则化更新机制。

### 6.3 第三优先级
- `profile_bucket`
- 模型增强标签

原因：
- 这部分会提升回答质量，但不是第一轮 MVP 的运行前提。

## 7. 建议的采集方式

### 7.1 Batch First
- 所有共享数据优先做成批次包，而不是实时散拉。
- 每轮 batch 必须有版本号、时间窗、条数和采样说明。

### 7.2 Structured First
- 优先下发结构化字段、标签字段、摘要字段。
- 原始文本和原始日志只保留必要样本。

### 7.3 Triggered Update
- `trend_signal`：按固定时间窗 + 热点事件触发
- `compliance_rule`：按月检查 + 政策事件触发
- `ingredient_knowledge`：按标准库修订或专题补充触发

## 8. 建议的 batch 设计

### 8.1 Base Batch
- 稳定商品样本
- 稳定成分知识
- 稳定法规底座

### 8.2 Focus Batch
- 某一趋势专题
- 某一类目专题
- 某一风险或合规专题

### 8.3 Eval Batch
- 独立保留的 `P2`
- 不参与训练，只参与对比评测

## 9. 对工程协作的意义
- 工程 A 需要稳定的 batch 和字段，才能做好 loader、retrieval、index
- 工程 B 需要稳定的 evidence 供给，才能做好 grounded QA
- 测试同学需要稳定的 batch version，才能做回测和对比
- 报告同学需要稳定的更新说明，才能写出可信的阶段总结

## 10. 对架构与数据 Owner 的直接建议
- 把“采集策略”看作接口设计，而不是看作纯数据运营
- 每次 batch 更新都要回答四个问题：
- 这次更新了哪些源
- 为什么更新
- 更新后影响哪几张表
- 哪些口径发生了变化

## 11. 当前最值得优先投入的动作

2. 补 `trend_signal` 的 `7d/30d` 双窗口口径
3. 把 `compliance_rule` 从第一版结构化推进到义务 taxonomy
4. 给 `ingredient_knowledge` 增加证据等级与风险标签统一规则
5. 准备第一版 `P2` holdout batch

## 12. 结论
- 是的，应该单独针对数据层做“数据源 -> 采集策略”的分析。
- 这不是补充性工作，而是当前 pipeline 的关键上游。
- 对你负责的部分来说，这份策略文档实际上就是后续 batch 管理、evaluation gate 和 distillation loop 的上游控制面。
