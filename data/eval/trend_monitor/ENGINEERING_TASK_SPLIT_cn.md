# 趋势监控工程任务拆分建议

这份文档给工程同学一个更具体的落地方案，目标是把当前这套趋势输入真正接成可持续运行的监控链路。

当前输入文件：
- `2026-04-13-report-registry.csv`
- `2026-04-13-trend-keyword-seed.csv`
- `README_cn.md`

## 目标

第一阶段不追求“全自动趋势分析平台”。
先把下面这条链路跑通：

- 读取报告登记表
- 读取关键词种子表
- 自动生成平台采集任务
- 采集结果落成统一结构
- 支持人工复核和后续增量扩词

这样做的价值是：
- 先验证这套输入是否能稳定产出信号
- 先把数据结构定住，后面换平台、加平台都不会返工太大
- 先把高优先级词跑起来，避免一开始工程量过宽

## 建议的工程模块

建议拆成 5 个模块。

### 1. 输入加载模块
负责：
- 读取 `report-registry.csv`
- 读取 `trend-keyword-seed.csv`
- 做字段校验
- 输出内部统一对象

最低要求：
- 校验必填字段
- 识别 `suggested_platforms` 和 `query_variants` 的 `|` 分隔格式
- 能过滤 `priority`、`confidence`、`signal_period_type`

建议输出对象：
- `report`
- `keyword_seed`
- `collection_task`

### 2. 任务生成模块
负责：
- 按平台拆任务
- 按 query 变体展开任务
- 给每个任务加调度优先级和抓取目的

推荐规则：
- `priority=high` 进第一批任务
- `confidence=high` 默认允许自动跑
- `risk_flag=high` 单独打标签，后续走风控监控链路
- `extractability=low` 的报告只保留一级主题，不自动扩更多细词

建议任务主键：
- `task_id`
- `keyword_id`
- `platform`
- `query`
- `run_batch`

### 3. 平台采集模块
负责：
- 按平台执行查询
- 返回统一格式结果
- 保留来源信息，方便回查

平台适配器：
- `douyin_adapter`
- `xiaohongshu_adapter`
- `taobao_adapter`
- `industry_news_adapter`

适配器接口建议统一：
- 输入：`platform`, `query`, `keyword_id`, `crawl_goal`
- 输出：标准结果记录列表

第一阶段不要把适配器做太重。
只要能稳定返回：
- 抓到了什么
- 来源是什么
- 热度或互动大概怎样
- 是否需要人工复核

### 4. 标准化落库模块
负责：
- 把各平台结果转成同一结构
- 去重
- 保留时间和来源
- 方便后续分析和复查

建议最少字段：
- `run_date`
- `keyword_id`
- `keyword`
- `normalized_keyword`
- `topic_cluster`
- `platform`
- `query`
- `source_type`
- `source_title`
- `source_url`
- `observed_signal`
- `engagement_or_heat`
- `risk_flag`
- `capture_note`

建议第一阶段先落 CSV 或 SQLite。
先不要一上来接复杂数据库，除非团队已经有统一数据基础设施。

### 5. 质检与回查模块
负责：
- 抽检抓取结果是否和关键词目标一致
- 识别无效 query、误抓 query、风险 query
- 反向更新关键词种子表

建议至少做三件事：
- 无结果率统计
- 明显噪声结果统计
- 高风险词单独复核列表

## 更具体的任务卡拆法

### 任务 A：输入解析和任务生成
交付物：
- 可读取两张 CSV
- 能输出展开后的平台任务表
- 支持按 `priority` / `confidence` 过滤

完成标准：
- 输入 1 个关键词，若有 2 个平台、3 个 query 变体，能正确展开成 6 条任务
- 输出中保留 `keyword_id`、`platform`、`query`

### 任务 B：平台采集适配器第一版
交付物：
- 至少先接 `bilibli` 和 `xiaohongshu`
- 每个平台能根据任务输入返回统一结构结果

完成标准：
- 能跑通 5 到 10 个高优先级词
- 每条结果都有 `source_url/source_id`
- 能区分“无结果”和“抓取失败”

### 任务 C：标准化结果表
交付物：
- 统一结果 schema
- 去重逻辑
- 本地存储方案

完成标准：
- 不同平台的数据能汇总到一张总表
- 同一 `keyword_id + platform + source_url` 不重复写入

### 任务 D：质检与人工回查支持
交付物：
- 无结果任务列表
- 高风险词结果列表
- 明显噪声结果列表

完成标准：
- 能快速回答两个问题：
  - 哪些词值得继续抓
  - 哪些词需要改 query 或下线

### 任务 E：增量扩词机制
交付物：
- 从结果里反推候选新词
- 输出人工确认候选表

完成标准：
- 不自动写回种子表
- 先输出候选词列表给 PM/研究同学确认

## 人员配置建议

### 方案 1：只有 1 位工程同学
建议顺序：
1. 先做输入解析 + 任务生成
2. 再做 `douyin` / `xiaohongshu` 两个平台
3. 再做统一结果表和质检脚本
4. 最后再接 `taobao` / `industry_news`

这个方案最稳，适合先验证整条链路。


这个方案很适合你现在的阶段。
因为目前核心难点不只是“怎么抓”，更是“先抓什么、哪些词值得抓”。

## 第一阶段建议范围

不要一上来做全覆盖。
建议第一阶段只跑下面这些：

- 平台：`douyin`、`xiaohongshu`
- 类型：`ingredient`、`claim`、`risk_compliance`
- 过滤条件：`priority=high` 且 `confidence=high`

原因：
- 这批词更能体现趋势监控价值
- 信号密度相对更高
- 更容易看出 query 是否需要改写

## 第一阶段建议里程碑

### M1：任务生成可跑
完成标准：
- 两张 CSV 能读
- 平台任务能展开
- 输出任务表可人工检查

### M2：双平台采集打通
完成标准：
- `douyin`、`xiaohongshu` 能各跑一批样本词
- 能落标准化结果

### M3：人工回查闭环
完成标准：
- 能标记无结果词、低质量词、高风险词
- 能产出一版“保留 / 改写 / 下线”的回查建议

## 工程上要避免的坑

- 不要把低可提取报告自动扩成很多细词
- 不要不同平台各自定义一套结果字段
- 不要把 `query_variants` 当归一化标签直接入库
- 不要一开始就做复杂评分模型
- 不要把自动扩词直接写回种子表

## 建议的协作边界

工程负责：
- 任务生成
- 平台采集
- 结果标准化
- 基础质检脚本

PM / 研究同学负责：
- 维护种子词
- 决定优先抓哪些主题
- 审核候选扩展词
- 判断哪些信号真有业务价值

这个边界比较清楚，也最适合当前团队推进节奏。
